# CLAUDE.md — Agentic RPG

## Project Overview

A browser-based, single-player RPG where narrative, world state, and character interactions are dynamically generated by LangChain-powered LLM agents orchestrated with LangGraph. Python/FastAPI backend, Next.js frontend, PostgreSQL database.

## Architecture

- **Backend**: Python 3.12+, FastAPI, Pydantic v2, asyncpg, Alembic, LangChain, LangGraph
- **Frontend**: Next.js 14+ (App Router), TypeScript, Tailwind CSS
- **Database**: PostgreSQL 16
- **Orchestration**: Docker Compose

See `docs/prd.md` for full PRD, `docs/roadmap.md` for build plan, `docs/specs/` for system specs, `docs/tech/` for technology decisions.

## Testing Standards

### Coverage Requirements

**All coverage metrics must meet or exceed 85%.** This applies to both backend and frontend.

| Metric | Minimum | Target |
|--------|---------|--------|
| Lines | 85% | 90%+ |
| Branches | 85% | 90%+ |
| Functions | 85% | 90%+ |
| Statements (frontend) | 85% | 90%+ |

These are hard requirements, not advisory. Tests that don't meet these thresholds are not done.

### Backend Coverage Configuration

Coverage is configured via `pyproject.toml`:

```toml
[tool.pytest.ini_options]
asyncio_mode = "auto"

[tool.coverage.run]
source = ["agentic_rpg"]

[tool.coverage.report]
fail_under = 85
show_missing = true
exclude_lines = [
    "pragma: no cover",
    "if TYPE_CHECKING:",
    "if __name__ ==",
]
```

Run: `uv run pytest --cov=agentic_rpg --cov-branch --cov-report=term-missing`

### Frontend Coverage Configuration

Coverage is configured via `jest.config.js`:

```js
coverageThreshold: {
  global: {
    branches: 85,
    functions: 85,
    lines: 85,
    statements: 85,
  },
},
```

Run: `npm run test -- --coverage`

### Red/Green TDD Workflow

This project uses red/green TDD as the primary development methodology:

1. **RED**: Write a failing test that describes the desired behavior
2. **GREEN**: Write the minimum code to make the test pass
3. **Move on**: No refactor phase — speed over elegance. Refactor later if needed.

Every implementation task must write tests BEFORE writing production code. Tests are the spec.

### What Makes a Good Test

Tests must have **specific, meaningful assertions**. A test should fail if the implementation is wrong and pass only if the implementation is correct.

**Good assertions:**
```python
# Python
assert result.name == "Aldric"
assert result.stats["health"] == 80
assert len(inventory.items) == 3
assert event.event_type == "character.stat_changed"
assert response.status_code == 200
assert "session_id" in data
```

```typescript
// TypeScript
expect(result.name).toBe("Aldric");
expect(result.stats.health).toBe(80);
expect(inventory.items).toHaveLength(3);
expect(response.status).toBe(200);
```

**Bad assertions (NEVER use these as the sole assertion in a test):**
```python
# Python — BANNED as sole assertion
assert result           # bare truthy check
assert result is not None  # too vague
assert True             # meaningless
```

```typescript
// TypeScript — BANNED as sole assertion
expect(result).toBeTruthy();    // passes for any truthy value
expect(result).toBeDefined();   // passes for anything non-undefined
expect(true).toBe(true);        // meaningless
```

### Test Organization

**Backend** (pytest):
- `tests/test_models/` — Pydantic model validation tests
- `tests/test_state/` — State manager CRUD tests (needs Postgres)
- `tests/test_events/` — Event bus (test_bus.py), schemas (test_schemas.py), persistence (test_persistence.py)
- `tests/test_tools/` — Tool registry + game tool unit tests (character, inventory, world, narrative)
- `tests/test_agent/` — Graph, context, prompt, story engine tests (mock LLM)
- `tests/test_llm/` — LLM client factory tests
- `tests/test_api/` — Health, sessions, game state, WebSocket, wiring, fixtures smoke tests
- `tests/conftest.py` — Shared fixtures (DB pool, sample data, mock LLM, event bus)

**Frontend** (Jest + React Testing Library):
- `__tests__/app/` — Page tests (home, layout, new, play)
- `__tests__/components/` — All 6 component tests
- `__tests__/lib/` — API client, Zustand store, WebSocket client tests
- `__tests__/types/` — Type validation tests
- `__tests__/integration/` — WebSocket + store integration tests

### Test Categories (Makefile)

Tests are split by DB dependency:

- **Unit tests** (`make test-unit`, no DB): test_models, test_tools, test_agent, test_llm, test_events/test_bus.py, test_events/test_schemas.py, test_api/test_health.py, test_api/test_websocket.py
- **DB tests** (`make test-db`, needs Postgres): test_state, test_events/test_persistence.py, test_api/test_sessions.py, test_api/test_game_state.py, test_api/test_fixtures_smoke.py, test_api/test_wiring.py
- **Frontend tests** (`make test-frontend`): All `__tests__/` files via Jest

Use `make test` for quick feedback (unit + frontend), `make test-all` for full suite.

### What to Test

- **Every public function/method** in the backend
- **Every tool** — correct state modification, event emission, error cases, edge cases
- **Every API endpoint** — success case, validation errors, not found, error responses
- **WebSocket** — connection, message handling, streaming, disconnection
- **Agent graph** — with mocked LLM, test tool call loops, error recovery, context assembly
- **Pydantic models** — serialization round-trips, validation errors on bad data
- **Frontend components** — rendering with props, user interactions, state updates
- **Frontend lib/** — API client calls, WebSocket message parsing, store actions

### What NOT to Test

- `__init__.py` files
- Imports and re-exports
- Framework boilerplate (FastAPI app factory internals, Next.js page wrappers)
- Type definitions
- Generated code

## Subagent Templates

### Test Quality Verifier

Use the `test-quality-verifier` subagent (defined in `~/.claude/agents/test-quality-verifier.md`) to audit test quality. **Run this subagent 3 times per phase** — once after writing tests, once after implementation, and once as a final check before committing.

#### How to invoke

Spawn the subagent with a prompt that specifies the scope and the project's thresholds:

```
Audit tests in {directory}. This project requires 85% minimum on ALL coverage
metrics (lines, branches, functions). Use these thresholds, not advisory defaults.

Specifically check for:

**Pass 1 — Vague Assertions:**
- Find every test with only bare truthy assertions (assert result, toBeTruthy, toBeDefined, assertIsNotNone as sole check)
- Find tests with no assertions at all
- Find tests that would pass even if the implementation returned a hardcoded stub
- Fix every vague assertion with a specific value/shape check

**Pass 2 — Coverage Gaps:**
- Run coverage: {coverage command}
- Identify uncovered lines, branches, and functions
- Add tests for uncovered error paths, edge cases, and branches
- Priority: error handling paths, conditional branches, boundary conditions

**Pass 3 — Test Strength:**
- For each test, ask: "Would this test catch a bug if the implementation was subtly wrong?"
- Check that tool tests verify both state changes AND event emissions
- Check that API tests verify response body contents, not just status codes
- Check that model tests verify validation rejects bad data, not just accepts good data
- Add negative test cases where missing (bad input, not found, unauthorized, edge cases)

After all 3 passes, output the Test Quality Report.
```

#### When to run

| Timing | What to check | Why |
|--------|--------------|-----|
| After writing tests (RED phase) | Pass 1: assertion quality | Catch vague tests before writing implementation |
| After implementation (GREEN phase) | Pass 2: coverage gaps | Ensure coverage meets 85% threshold |
| Before committing | Pass 3: test strength | Final audit — tests must catch real bugs |

## Code Style

### Python
- Use `ruff` for linting and formatting
- Type hints on all public functions
- Docstrings on public classes and non-obvious functions
- Pydantic models: Field descriptions on every field, StrEnum for fixed value sets
- async/await everywhere — no sync DB calls

### TypeScript
- Strict mode enabled
- Interfaces for all data structures (matching backend Pydantic models)
- Tailwind utility classes only — no custom CSS
- `any` is acceptable for speed, but prefer proper types where easy

### Git
- Commit after each phase completes
- Descriptive commit messages: `"phase N: brief description"`
- Each Ralph works on its own branch, merge to main when done
